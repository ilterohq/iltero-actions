# Configuration

Iltero Actions is configuration-driven. Nearly all behavior — which stacks exist, which environments they map to, what policies they enforce — is declared in a `config.yml` file that lives alongside your IaC code. The examples and file structure below show Terraform (the currently supported IaC tool); OpenTofu, Pulumi, and CloudFormation are on the roadmap. This page covers:

- [Repository Layout](#repository-layout) — where things go
- [Self-Contained Units](#self-contained-units) — the Terraform module structure we require
- [Stack Configuration](#stack-configuration) — greenfield `config.yml` reference
- [Brownfield (Attached) Stacks](#brownfield-attached-stacks) — for existing Terraform projects

## Repository Layout

A typical repo using Iltero Actions looks like this:

```text
your-repo/
├── .github/
│   └── workflows/
│       └── iltero.yml                 # Generated by Iltero on repo init
├── infra/
│   └── stacks/
│       ├── network/
│       │   ├── config.yml             # Network stack configuration
│       │   └── units/
│       │       ├── vpc/
│       │       └── subnets/
│       └── database/
│           ├── config.yml             # Database stack configuration
│           └── units/
│               └── rds/
└── README.md
```

Each stack is a directory with its own `config.yml` and one or more **units**. A unit is a self-contained Terraform module that deploys independently.

## Self-Contained Units

**Each infrastructure unit must be self-contained** with all required Terraform files. This eliminates runtime file assembly: what you see in git is exactly what deploys.

### Required Structure

```text
infra/stacks/my-stack/
├── config.yml                         # Stack configuration
└── units/
    └── network-baseline/
        ├── main.tf                    # Module calls and resources
        ├── providers.tf               # Provider configurations (with workspace tags)
        ├── versions.tf                # Terraform and provider version constraints
        ├── backend.tf                 # Backend configuration (partial config)
        ├── data.tf                    # Data sources (optional)
        └── config/                    # Environment-specific configuration
            ├── development.tfvars
            ├── staging.tfvars
            ├── production.tfvars
            └── backend/
                ├── development.hcl
                ├── staging.hcl
                └── production.hcl
```

### File Purposes

| File | Purpose |
|------|---------|
| `main.tf` | Module calls, resources, locals |
| `providers.tf` | Provider configuration with `default_tags` including workspace/environment |
| `versions.tf` | Terraform version constraints and required providers |
| `backend.tf` | Backend configuration using `-backend-config` partial config |
| `data.tf` | Data sources for cross-unit dependencies (optional) |
| `config/{env}.tfvars` | Environment-specific variable values |
| `config/backend/{env}.hcl` | Environment-specific backend settings (bucket, key, etc.) |

### Why Self-Contained?

1. **No Runtime Assembly** — no copying files from `environments/` folders at runtime
2. **Predictable** — what's in git is what deploys
3. **IDE Support** — full IntelliSense and validation in each unit folder
4. **Independent** — each unit can be deployed, tested, and validated in isolation
5. **Auditable** — clear history of all changes in one place

## Stack Configuration

All stack behavior is controlled by `config.yml`:

```yaml
version: 1.0.0

stack:
  id: 0b278217-a809-465a-b9df-00eda8414cb8
  name: my-infrastructure
  slug: my-infrastructure

terraform:
  version: 1.5.7

infrastructure_units:
  - name: network-baseline
    path: units/network-baseline
    enabled: true
    depends_on: []
  - name: security-baseline
    path: units/security-baseline
    enabled: true
    depends_on: [network-baseline]

environments:
  development:
    git_ref:
      type: branch
      name: develop                    # develop → development
    compliance:
      enabled: true
      scan_types: [static, evaluation]
    security:
      severity_threshold: medium

  production:
    git_ref:
      type: branch
      name: main                       # main → production
    compliance:
      enabled: true
      scan_types: [static, evaluation]
    security:
      severity_threshold: high
    deployment:
      require_approval: true
```

### Key Sections

| Section | Purpose |
|---------|---------|
| `stack` | UUID, name, slug — identifies the stack in Iltero |
| `terraform.version` | Terraform version used for plan/apply |
| `infrastructure_units` | Ordered list of units with `depends_on` edges |
| `environments.<env>.git_ref` | Branch-to-environment mapping (e.g., `main` → `production`) |
| `environments.<env>.compliance` | Which scan types run (`static`, `evaluation`) |
| `environments.<env>.security.severity_threshold` | Findings at this severity or above fail the build |
| `environments.<env>.deployment.require_approval` | Whether GitHub Environment approval is required |

### git_ref Types

- `branch` — exact branch name match (e.g., `main`, `develop`)
- `tag` — exact tag name match
- `pattern` — fully-anchored regex (e.g., `release/v\d+\.\d+`)

Iltero Actions **fails closed** if no `git_ref` matches the current branch — there is no silent fallback to an arbitrary environment. For pull requests, the environment is resolved from the **base branch** (the merge target), so compliance runs under the policies of the environment the code will land in.

See [`schemas/config.schema.json`](../schemas/config.schema.json) for the complete JSON Schema, and [`examples/config.example.yml`](../examples/config.example.yml) for a fully-commented reference config.

## Brownfield (Attached) Stacks

For existing Terraform projects where you can't restructure into self-contained units, use a **brownfield** config at `.iltero/config.yml` in your repo root.

```yaml
stack:
  id: a1b2c3d4-e5f6-7890-abcd-ef1234567890
  name: my-existing-project
  type: attached                       # Required for brownfield
  terraform_working_directory: terraform  # Path to existing TF root

environments:
  production:
    git_ref:
      type: branch
      name: main
    compliance:
      enforcement_mode: permissive     # Report violations without blocking
      policy_sets: [aws-security-baseline]
```

### Differences from Greenfield

| Aspect | Greenfield | Brownfield |
|--------|-----------|------------|
| Config location | `infra/stacks/<stack>/config.yml` | `.iltero/config.yml` |
| `stack.type` | (not set) | `attached` |
| Units | Listed in `infrastructure_units` | Not used — the working directory *is* the unit |
| Workflow trigger paths | `infra/stacks/**` | `**/*.tf`, `.iltero/**` |
| Pipeline input | `stacks_path: infra/stacks` | `stacks_path` omitted, `config_path` defaults to `.iltero/config.yml` |

### Gradual Adoption

Brownfield stacks often default to `enforcement_mode: permissive` so compliance failures report but don't block deployment. Once the existing codebase is clean, flip to `strict` (the default).

See [`examples/brownfield.yml`](../examples/brownfield.yml) and [`examples/config.brownfield.example.yml`](../examples/config.brownfield.example.yml) for complete references.
